# terraform-provider-panther

Terraform provider for Panther (panther-labs/panther). Written in Go using the HashiCorp Terraform Plugin Framework.

## Project Structure

- `main.go` — Provider entrypoint, goreleaser sets version
- `internal/provider/` — Resource implementations (schema, CRUD)
  - `resource_s3_source.go` — S3 log source (GraphQL API)
  - `httpsource_resource.go` — HTTP log source (REST API)
  - `resource_httpsource/httpsource_resource_gen.go` — Generated code from `generator_config.yml`
- `internal/client/` — API client interfaces and types
  - `panther/` — GraphQL and HTTP client implementations
- `examples/` — Terraform example configs (used by doc generator)
- `docs/` — Generated documentation (do not edit manually)

## Key Commands

- **Build:** `go build -v ./...`
- **Test:** `go test -v ./...`
- **Generate docs + format examples:** `go generate ./...` (requires `terraform` CLI installed)
- **Acceptance tests** require `PANTHER_API_URL` and `PANTHER_API_TOKEN` env vars

## Development Patterns

- Uses **Terraform Plugin Framework** (not the older SDK)
- Two API patterns coexist:
  - **GraphQL** (`go-graphql-client`) for S3 sources — client in `internal/client/panther/panther.go`
  - **REST** for HTTP sources — client in `internal/client/panther/panther.go`
  - HTTP auth client in `internal/client/panther/http.go`
- Provider config: `url` + `token` (or env vars `PANTHER_API_URL`, `PANTHER_API_TOKEN`)
- Resource type prefix is `panther_` (e.g. `panther_s3_source`, `panther_httpsource`)

## Code Generation

- Resource schemas are partially generated from `generator_config.yml` + the Panther OpenAPI spec
- `provider-code-spec.json` is auto-generated by `tfplugingen-openapi` — do not edit manually
- `resource_*/..._resource_gen.go` files are auto-generated by `tfplugingen-framework` — do not edit manually
- After changing generated code or examples, run `go generate ./...` and verify `git diff --exit-code` (CI checks this)

## CI

- GitHub Actions workflow (`.github/workflows/go.yml`) runs on PRs: generate check, build, test
- Releases via goreleaser (`.goreleaser.yml`)

## Conventions

- Apache 2.0 license headers on all Go files
- Resource models use `tfsdk` struct tags
- Client types use `graphql` tags (GraphQL) or `json` tags (REST)
- Fields that force resource recreation use `stringplanmodifier.RequiresReplace()`

## Adding a New REST API Resource

New REST resources use two code-generation tools for scaffolding and schema generation. The reference implementation is `panther_httpsource`. The resource must already exist in the Panther REST API with CRUD endpoints.

### Prerequisites

Install the codegen tools if you don't already have them:

```sh
go install github.com/hashicorp/terraform-plugin-codegen-framework/cmd/tfplugingen-framework@latest
go install github.com/hashicorp/terraform-plugin-codegen-openapi/cmd/tfplugingen-openapi@latest
```

The binaries install to `$(go env GOBIN)` (or `$(go env GOPATH)/bin`). If the tools aren't on your PATH (e.g. when using asdf), invoke them with:

```sh
"$(go env GOBIN)/tfplugingen-framework"
"$(go env GOBIN)/tfplugingen-openapi"
```

Download the Panther OpenAPI spec (`panther_open_api_v3_spec.yaml`) from the Panther docs:
https://docs.panther.com/panther-developer-workflows/api/rest#openapi-specification

### Step 1: Scaffold the resource

```sh
tfplugingen-framework scaffold resource \
  --name {resource_name} \
  --output-dir ./internal/provider
```

This creates the resource file stub at `internal/provider/{resource_name}_resource.go` and the generated package at `internal/provider/resource_{resource_name}/`.

### Step 2: Update `generator_config.yml`

Add the new resource with its REST endpoint paths and methods. Follow the existing `httpsource` entry as a template:

```yaml
resources:
  {resource_name}:
    create:
      path: /your-resource-path
      method: POST
    read:
      path: /your-resource-path/{id}
      method: GET
    update:
      path: /your-resource-path/{id}
      method: PUT
    delete:
      path: /your-resource-path/{id}
      method: DELETE
    schema:
      ignores:
        - integrationId  # fields to exclude from generated schema
```

### Step 3: Generate the provider code spec from OpenAPI

`provider-code-spec.json` already exists at the repo root (it contains specs for existing resources). Running this command **updates** it — the new resource is added alongside existing ones:

```sh
tfplugingen-openapi generate \
  --config ./generator_config.yml \
  --output ./provider-code-spec.json \
  {path_to_downloaded_openapi_yml}
```

This reads the OpenAPI spec and `generator_config.yml` to produce/update `provider-code-spec.json`, which describes the resource schema.

### Step 4: Generate the resource model and schema

```sh
tfplugingen-framework generate resources \
  --input ./provider-code-spec.json \
  --output ./internal/provider
```

This generates/updates `internal/provider/resource_{resource_name}/{resource_name}_resource_gen.go` containing the schema function and model struct. **Do not edit this generated file** — it will be overwritten.

### Step 5: Implement CRUD methods

Fill in the scaffolded resource file at `internal/provider/{resource_name}_resource.go`. Use `httpsource_resource.go` as the reference:

- **Schema overrides**: The generated schema doesn't support everything. In your `Schema` method, call the generated schema function then manually add:
  - **ID handling depends on whether the ID is server-generated or user-provided:**
    - **Server-generated ID** (e.g. httpsource): `id` is `Computed: true` + `UseStateForUnknown()` plan modifier — the server assigns the ID on create
    - **User-provided ID**: `id` is `Required: true` + `RequiresReplace()` plan modifier — the user sets the ID, changing it forces recreation
  - `stringdefault.StaticString("")` on optional string fields (the generator makes them Optional+Computed, so without defaults they'll be "unknown")
  - `booldefault.StaticBool(false)` on optional bool fields
  - `listdefault.StaticValue(emptyList)` on optional list fields
  - `mapdefault.StaticValue(emptyMap)` on optional map fields
  - `objectdefault.StaticValue(types.ObjectNull(...))` on optional nested objects
  - Note: the generator sometimes adds defaults itself (e.g. `int64default` with validators) — check the generated schema before adding your own
- **Configure**: Extract `*panther.APIClient` from `req.ProviderData`, assign `r.client = c.RestClient`
- **Create**: Read plan -> map model to client input -> call API -> set `data.Id` from response (for server-generated IDs) -> save state
- **Read**: Read state -> call Get API -> map response to model (skip sensitive fields the API returns as empty) -> save state
- **Update**: Read plan -> map to update input (include Id) -> call API -> save state
- **Delete**: Read state -> call Delete API
- **ImportState**: `resource.ImportStatePassthroughID(ctx, path.Root("id"), req, resp)` (not generated, must be added manually)

### Step 6: Add client types and REST methods

1. **`internal/client/panther.go`**: Define request/response structs with `json` tags. Add methods to the `RestClient` interface.
2. **`internal/client/panther/panther.go`**: Implement the CRUD methods on `RestClient` following the HTTP source pattern:
   - Add a path constant (e.g. `const RestFooPath = "/your-resource-path"`)
   - Expected HTTP status codes vary per resource — check the API docs (e.g. httpsource POST returns `201`)
   - All requests use `http.NewRequestWithContext` and the shared `Doer` (adds `X-API-Key` header)

### Step 7: Register, add examples, and generate docs

1. Register in `internal/provider/provider.go` -> add constructor to `Resources()` slice
2. Add `examples/resources/panther_{resource_name}/resource.tf` (used by doc generator)
3. Add acceptance tests in `internal/provider/{resource_name}_resource_test.go`
4. Run `go generate ./...` to regenerate `docs/` and format examples
5. Verify: `go build -v ./...` and `go test -v ./...`

## Updating an Existing REST Resource

For schema changes or new attributes on an existing resource:

1. Update `generator_config.yml` if REST endpoints changed
2. Re-run Step 3 (openapi generate) and Step 4 (framework generate) from above with the latest OpenAPI spec
3. Update the Schema overrides in the resource file if new optional fields need defaults
4. Update client types/methods if the API payloads changed
5. Run `go generate ./...` and verify

## Code Generation Limitations

The generators don't cover everything. These must be handled manually in the resource's `Schema` method:
- **`UseStateForUnknown`** plan modifier on server-generated `id` -- not supported by generator
- **`RequiresReplace`** plan modifier on user-provided `id` -- not supported by generator
- **Default values** for most Optional+Computed fields -- generator doesn't set them (exception: `int64` fields with validators sometimes get defaults)
- **`ImportState`** method -- not generated, must be implemented manually
- **Sensitive field handling in Read** -- API returns sensitive fields as empty strings; don't overwrite state for those

### Generated schema surprises

The generated schema may include fields you didn't expect or model types differently than anticipated. Always inspect the generated `_resource_gen.go` before writing the resource implementation:
- **Extra fields**: The OpenAPI spec may expose server-managed fields that need defaults or should be added to `schema.ignores` in `generator_config.yml`
- **Nested objects generate custom types**: `SingleNestedAttribute` and `ListNestedAttribute` generate custom `Type`/`Value` structs. Use these custom types when converting to/from the model — don't use `types.Object` directly
- **Complex nested types may differ from expectations**: e.g. a nested object might generate as `list(map(string))` rather than `list(object)` depending on how the OpenAPI spec defines it. Check the generated types before implementing conversion logic
- **`Computed`-only fields**: Pure `Computed` fields are set by the server and don't need defaults. But `Optional+Computed` fields do

## Key Gotchas

- **Optional+Computed fields**: Without explicit defaults, Terraform treats missing values as "unknown" causing perpetual diffs. Always set `stringdefault.StaticString("")` or equivalent.
- **Nested objects**: When the API always returns a non-nil nested object, check if inner fields are actually populated before setting in state (see `logStreamTypeOptions` handling in both resources).
- **REST client base URL**: `CreateAPIClient` strips `/public/graphql` for backwards compatibility. `RestClient.url` stores the base URL only — each REST method prepends its own path constant (e.g. `RestHttpSourcePath`).
- **User-provided IDs**: When the resource ID is user-provided (not server-generated), use `Required: true` with `RequiresReplace()` instead of `Computed: true` with `UseStateForUnknown()`. In `Create`, the ID is already in the plan — no need to set it from the response.
- **Query parameters on REST endpoints**: Some endpoints need query parameters. Check the API docs for required query params on create/update.
- **Async resources**: If the underlying resource has async provisioning (e.g. Firehose), deletion may fail right after creation. See the manual delete retry loop in `httpsource_resource_test.go`.
- **Use `*bool` and `*int64` for optional fields in client types**: This ensures `omitempty` works correctly with JSON marshaling (Go's zero values for `bool` and `int64` would be omitted otherwise).
